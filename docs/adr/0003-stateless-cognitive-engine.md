# ADR-0003: ステートレス認知エンジンへの転換

- **Status:** Accepted
- **Date:** 2026-04-10
- **Context:** akc-mcp のユースケース検討で、サーバー側に状態を持つ設計の問題が明らかになった

## 問題

現在の akc-mcp はサーバー側にデータディレクトリ（`~/.config/akc/`）を持ち、エピソードログ・知識・スキル・ルール・identity・constitution を管理している。この設計には以下の問題がある:

1. **エピソードの入力元が不明** — `record_episode` を誰がいつ呼ぶのかが定義されていない
2. **蒸留結果がクライアントに反映されない** — サーバー側のファイルを更新しても、クライアントエージェントの行動は変わらない（ADR-0002）
3. **プラットフォーム依存** — `~/.config/akc/` の構造を前提としており、他環境で使いにくい
4. **マルチテナントの複雑さ** — ユーザーごとのデータ分離が必要になり、セキュリティリスクが増大

## 決定

akc-mcp を **ステートレスな認知エンジン** に転換する。サーバー側にはデータも状態も持たない。

### インターフェース

全コンテキストをクライアントが入力として渡し、変換結果を受け取る:

```
入力:
  episodes      — 活動ログ (JSONL)
  knowledge     — 蓄積済み学習パターン
  skills        — 既存スキル群
  rules         — 既存ルール群
  identity      — 現在のアイデンティティ
  constitution  — 憲法（行動原則）

出力:
  knowledge'    — 更新された学習パターン
  skills'       — 新規/更新スキル
  rules'        — 新規/更新ルール
  identity'     — 更新されたアイデンティティ
```

### 設計特性

- **純粋関数**: 同じ入力に対して同じ出力。副作用なし
- **状態ゼロ**: サーバー側にデータディレクトリ不要
- **プラットフォーム非依存**: JSONL と Markdown を渡せれば何からでも使える
- **認証不要**: ステートレスなので、LLM の API キーだけあれば動作（ユーザーの API キーで LLM 呼び出し）

### 不要になるもの

- `~/.config/akc/` データディレクトリ
- `record_episode` ツール（エピソード管理はクライアント側の責務）
- マルチテナント機構（ユーザーデータ分離）
- Fly.io リモートデプロイ（既に ADR-0001 で削除済み）

## 利点

1. **極めて高い汎用性** — Claude Code, Cursor, Codex, 自作エージェント、REST API 経由、何からでも利用可能
2. **セキュリティ簡素化** — サーバーにデータがないので漏洩リスクゼロ
3. **テスト容易性** — 入出力が明確な純粋関数
4. **contemplative-agent との正しい分離** — 認知「エンジン」としての切り出しが本来の意図に合致

## 未実装（次セッション以降）

- 既存の stateful 実装からの移行
- ツール API の再設計（引数として全コンテキストを受け取る形式）
- 入力サイズの制限・分割戦略（大量のエピソードやスキルを渡す場合）
- LLM コンテキストウィンドウとの兼ね合い
