# ADR-0002: MCP 認知エンジンの統合設計 — ブリッジ層は不要

- **Status:** Accepted (Supersedes initial "bridge layer required" draft)
- **Date:** 2026-04-10
- **Context:** akc-mcp を Claude Code 等の AI エージェントから利用する際の設計問題

## 問題

contemplative-agent の認知層を MCP ツールとして切り出した（akc-mcp）が、蒸留結果がクライアントエージェントの行動に反映されないのではないか、という懸念が生じた。

### 当初の懸念: 蒸留と行動の分離

- **contemplative-agent**: 蒸留も行動も同一プロセス。skills/, rules/, identity.md を自分で書いて自分で読む。サイクルが回る
- **akc-mcp**: 蒸留結果を MCP レスポンスとして返すだけ。クライアント側の skills/rules/identity には自動反映されない

Claude Code が `distill` を呼んでも、結果は `~/.config/akc/` に閉じ、Claude Code 自身の `~/.claude/skills/` には直接書き込まれない。

## 決定

**専用のブリッジ層は不要。** 蒸留結果を受け取るのは LLM エージェント自身であり、結果をどこに配置すべきか判断・実行する能力を持っている。

```
akc-mcp (認知エンジン — プラットフォーム非依存)
  ↓ 蒸留結果を構造化テキストで返す
LLM エージェント (Claude Code, Cursor, Codex 等)
  ↓ 自分で判断して適切な場所に配置
  ↓ 必要なら人間に承認を求める
各プラットフォームの設定:
  Claude Code → ~/.claude/skills/, ~/.claude/rules/
  Cursor      → .cursorrules
  Codex       → codex.md 等
```

### なぜこれで動くか

1. **エージェントはファイル操作能力を持つ** — 蒸留結果を受け取ったら、自分の設定ディレクトリに保存できる
2. **エージェントは自分の構造を知っている** — Claude Code は `~/.claude/skills/` の形式を理解している
3. **承認ゲートが自然に成立する** — エージェントがユーザーに「このスキルを保存しますか？」と聞くだけ
4. **プラットフォーム差異の吸収もエージェントが行う** — 形式変換のアダプターコードは不要

akc-mcp の役割は「構造化された蒸留結果を返す」ことに専念する。配置と反映はクライアントエージェントの責務。

## 未解決の課題

- エピソードの入力元（各プラットフォームの活動ログをどう record_episode に流すか）
- 蒸留結果の出力形式の標準化（エージェントが解釈しやすい形式）
- identity のセマンティクス（各プラットフォームの identity 概念との対応）

## 教訓

MCP クライアントが「ツールの結果を受け取る受動的な存在」だという前提が間違いだった。クライアントは LLM エージェントであり、結果を解釈し、判断し、行動する能力を持つ。認知エンジンは結果を返すだけで十分であり、配置の責務をエンジン側に持たせるのは過剰設計。
