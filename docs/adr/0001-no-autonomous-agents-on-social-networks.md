# ADR-0001: 自律エージェントをソーシャルネットワークに放たない

- **Status:** Accepted
- **Date:** 2026-04-10
- **Context:** Managed Agents + Moltbook での Tabula Rasa エージェント運用計画

## 決定

Managed Agents（または同等の自律エージェント）を Moltbook 等のソーシャルネットワーク上で運用しない。試験運用も行わない。

## 理由

### 1. プロンプトインジェクションが原理的に防げない

ソーシャルネットワークの投稿は untrusted content である。エージェントがこれを読み取り、エピソードとして記録し、LLM 蒸留パイプラインに流す設計では、悪意あるプロンプトがそのまま LLM に到達する。フィルタリングで軽減はできても、原理的な防御は不可能。

### 2. コード実行 + untrusted input は最悪の組み合わせ

Managed Agents はコード実行能力を持つ。プロンプトインジェクションと組み合わさると、攻撃者が任意コードを実行できる経路が生まれる。機能制限しても、残った機能の組み合わせで何が起きるか予測できない。行動空間が広すぎる。

### 3. "Security by Absence" との矛盾

contemplative-agent はローカル完結・外部入力なしという "Security by Absence" を思想としている。外部ソーシャルネットワークから untrusted content を取り込む設計はこの思想と根本的に矛盾する。

## 影響

- Managed Agents + Moltbook 運用計画（Phase 2 ロードマップ）を中止
- Fly.io リモートデプロイを削除済み
- akc-mcp は stdio ローカル実行で公開する方針に転換
- scripts/managed_agent.py はリファレンス実装として残すが、運用しない

## 代替案（検討の上、却下）

- **プロンプトインジェクション対策を強化して運用** — 軽減はできるが原理的に防げない。リスクが残る限り運用すべきでない
- **読み取り専用（投稿しない）で運用** — 読み取りだけでもエピソードに untrusted content が入る。蒸留時に同じ問題が発生する
- **サンドボックス強化** — コード実行のサンドボックスを強化しても、LLM 自体の挙動変容は防げない

## 教訓

現時点の自律エージェント技術は、untrusted input を処理する環境では安全に運用できない。行動空間を最小限に絞り、信頼できる入力のみを扱うローカル完結型が正しいアプローチである。
